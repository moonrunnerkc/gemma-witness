# How Gemma.Witness compares to C2PA

Audience: an implementer, a security auditor, or a maintainer of a
C2PA-compatible tool who wants to understand what Gemma.Witness adds,
what it leaves out, and where the two ecosystems can be made to
interoperate. The short version of this comparison is in the README
section [§"How this compares to C2PA"](../README.md#how-this-compares-to-c2pa);
this document is the technical deep dive.

## What C2PA is

The Coalition for Content Provenance and Authenticity (C2PA) is a
multi-stakeholder body that publishes the Content Credentials
specification. Founding members include Adobe, Microsoft, the BBC, the
New York Times, Sony, Truepic, Arm, and Intel. The spec defines:

- A **manifest** structure embedded in media files (JPEG, PNG, MP4,
  PDF, etc.) carrying signed assertions about the file's origin and
  edit history.
- An **assertion vocabulary** with stable identifiers for common claims:
  `c2pa.actions` (edits applied to the asset), `c2pa.thumbnail.claim`,
  `c2pa.hash.data` (covered byte ranges), `c2pa.training-mining` (AI
  training data declaration), `c2pa.ai-generated`, `c2pa.creative-work`,
  and more.
- A **signing model** based on COSE (CBOR Object Signing and Encryption,
  RFC 8152) with X.509 certificate chains as trust anchors. Multiple
  signatures may chain across edits to record provenance through a
  workflow.
- A **trust-list** mechanism. Verifiers consult a list of trusted
  signer certificates, which in practice is fetched and cached
  out-of-band.

C2PA's reference implementation is `c2patool` and the Rust `c2pa`
crate; widely deployed verifiers include the Content Credentials browser
extension and the inspect.contentauthenticity.org web verifier.

## Where Gemma.Witness aligns with C2PA

Both systems treat tamper-evidence as a cryptographic property:

- **Hashing.** SHA-256 over raw asset bytes. C2PA spec v2.x §16; the
  Gemma.Witness manifest schema documents the same primitive in
  [`spec/bundle-format.md`](../spec/bundle-format.md).
- **Signed manifests.** C2PA uses COSE over CBOR; Gemma.Witness uses
  RFC 8785 JCS over JSON. Both produce a deterministic byte sequence
  the signer commits to.
- **Schema versioning.** A manifest version field gates routing through
  the verifier; older verifiers fail closed on unknown versions rather
  than guessing.
- **Public-key trust.** A verifier identifies a signer by public-key
  material rather than by metadata that can be spoofed.

If you implement a C2PA verifier and a Gemma.Witness verifier, most of
the crypto-primitive code can be shared. The divergence is at the layer
above: what is signed, what the assertions claim, and what the verifier
fetches.

## Where Gemma.Witness diverges

### 1. The reasoning trace is the artifact

C2PA's AI-related assertions are declarative: `c2pa.training-mining`
records that an asset was used for training; `c2pa.ai-generated` records
that AI was involved in producing the asset. Neither assertion binds
the AI's actual output or its intermediate reasoning. A C2PA-signed
image generated by an AI carries an authenticatable statement that AI
was involved, but the chain-of-thought that produced it is not
recoverable from the manifest.

Gemma.Witness flips this. The thinking-channel bytes Gemma 4 emits during
the structuring and consistency-check passes are written into the
bundle verbatim (no whitespace stripping, no pretty-printing, no
summarization) and committed to by the signature. A verifier opening a
bundle can read the model's chain-of-thought and judge whether the
structured incident report follows from the inputs. The model identity
is also part of the signature scope: `model_id`, `revision`, and the
SHA-256 of `model.safetensors` are pinned to the published Hugging Face
LFS oid. A swap of the model that produced the trace is detectable;
the verifier rejects the bundle when the fingerprint is not on the
embedded known-good list.

This matters when the reasoning is the evidence rather than incidental
metadata, which is the field-journalism and accountability use case
Gemma.Witness leads with.

### 2. Multi-asset bundles

A C2PA manifest is per-asset. One image carries its own manifest; an
audio file alongside that image carries a different manifest with no
cross-manifest binding. A workflow that produces an incident report
spanning audio plus N photos plus a structured summary has to either
(a) merge them into a single asset (PDF, MP4) so one manifest can cover
the whole, or (b) maintain external bookkeeping that the audio and the
images go together, with no cryptographic guarantee of that
correspondence surviving redistribution.

The `.witness` bundle is a ZIP whose manifest commits to:

- the audio file's raw-byte SHA-256,
- each image's raw-byte SHA-256,
- the reasoning-trace bytes,
- the structured incident report,
- the model fingerprint,
- a cross-modal consistency verdict ("the audio narration matches the
  photographs", "the photos contradict the audio at this specific
  point", etc.),
- the per-pass inference parameters (temperature, top_p, max_tokens,
  prompt SHA-256).

The consistency verdict is itself a signed assertion. Any individual
asset can be re-extracted and verified independently; the whole bundle
can be verified atomically.

### 3. Zero-network verifier

C2PA's trust-list mechanism is a list of accepted signer certificates,
typically fetched at install time and refreshed via a CDN. Revocation
relies on a CRL or OCSP query. The default implementation pattern is
network-touching, and even verifiers that ship a cached trust list are
usually not designed to operate offline indefinitely.

Gemma.Witness's verifier is a single 80 KB HTML file. The build script
inlines `known-fingerprints.json`, `trusted-signers.json`, and the
result of build-time Sigstore verification of the fingerprint registry
envelope. The build asserts no `fetch`, `XMLHttpRequest`, or
`importScripts` call survived bundling, and a CSP `default-src 'none'`
meta tag is required on the output. A reader who downloads the
verifier once can use it from a USB stick on an air-gapped machine
indefinitely.

Trust transfers via SHASUMS256.txt covering `verify.html`, signed by
the same cosign keyless OIDC identity documented in
[`RELEASE.md` §"Trust anchors"](../RELEASE.md#trust-anchors). The
build-time Sigstore verification of the fingerprint registry envelope
is the build-side proof; the runtime verifier surfaces the result as
the "Registry signature" check row. The rationale and the build-time vs
runtime trade-off are in the `apps/verifier/build-verify-registry.mjs`
header.

## Where neither defends

Both systems share the same epistemic blind spots:

- A reporter who chooses to sign a false statement signs the false
  statement; tamper-evidence cannot ask the reporter to be honest.
- A device compromised before sealing produces a bundle the verifier
  cannot distinguish from a clean capture, because the compromise
  precedes the signature.
- A microphone or camera tampered upstream of the file system writes
  whatever bytes the attacker chose and the device signs them.
- Social engineering of the trust-anchor lists works against any
  signing system; convincing a maintainer to merge a malicious key into
  `trusted-signers.json` or a C2PA root list defeats verification
  downstream.

The [`README.md` §"Threat model"](../README.md#threat-model) and the
C2PA spec's security considerations sections name these limits
explicitly. Picking a system means picking a threat model, not picking
who "wins"; the two systems are complementary for different audiences.

## Interoperation

A future direction worth naming: a `.witness` bundle could be packaged
as a C2PA manifest store for users who want their evidence to round-trip
through C2PA-aware tooling. The mapping is straightforward in one
direction (assertions translate, the model fingerprint becomes a custom
assertion under the `gemma.witness.*` namespace, the reasoning trace
becomes an embedded thumbnail-class blob). The other direction is
harder: a C2PA-only manifest does not carry a reasoning trace, and the
Gemma.Witness verifier would refuse a bundle whose manifest lacks the
fields the schema requires.

If a use case is "I have a C2PA workflow already and I want to add
Gemma 4's reasoning as a verifiable artifact," that adapter is a
plausible feature. File an issue if that maps to a real workflow you
own.

## Honest note on standardisation

C2PA is a standard with multi-vendor backing and an existing deployed
ecosystem. Gemma.Witness is a project-specific bundle format with a
single-maintainer governance model today. If standardisation matters
more to a user than the specific properties Gemma.Witness adds, C2PA
is the right pick. The promise this project makes is not "use this
instead of C2PA"; it is "if you need the sealed reasoning trace, the
multi-asset incident model, or the zero-network verifier and C2PA does
not yet offer them, use this in parallel."

## Open work

- Map the Gemma.Witness manifest fields onto a C2PA assertion vocabulary
  proposal so the two formats could merge cleanly.
- File a draft assertion definition under the `gemma.witness.*` prefix
  with the C2PA technical committee once the manifest is at v1.0 and the
  schema has stopped evolving.
- Track the C2PA AI-generated content addendum work; if a future revision
  binds the model's actual output rather than just naming AI
  involvement, this section gets shorter.
